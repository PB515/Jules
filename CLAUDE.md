## What this is

**Synergy** (formerly "Jules") is a mobile-first Progressive Web App that is the shared engagement platform for every club in the Adani University FMS/Infra Management MBA program (12-13 clubs, one shared system — not a single-club app). It turns club participation (meetings, expert sessions, volunteer tasks, live quizzes) into a gamified engagement system — "Joules" earned via QR check-in and quiz ("Surge") participation, standing tiers (Ember → Volt → Current → Plasma), and a permanent archive ("Catalyst Records"). **Three parts**: the student "Node" experience (mobile-installed-PWA only, decision 29 — never a laptop browser, by explicit professor requirement), the admin "Reactor Command Center" (role-scoped — being restructured into Student/Committee Member/Professor per decision 45, still Owner/Officer/Volunteer as of Phase 0), and a public "General" site (club info, event calendar, PWA install page, Event Reports — renamed from Afterglow, decision 45 — Gallery — no login, works on any device). This is an **authenticated app with private per-user data** for two of the three parts — the portal golden path applies, security-first.

**Naming note**: the app and its underlying code/folders are still named "Jules" in the repo path and a handful of internal identifiers (`lib/jules/*`, the `jules` git repo itself) — only user-visible copy and the product name were rebranded to Synergy in Phase 0 (decision 45). Do not rename internal folders/variables as part of later phases unless specifically asked; it's cosmetic-only churn with real regression risk for no user-facing benefit.

Full concept + already-decided product/technical detail: [`docs/project-spec.md`](docs/project-spec.md) (the canonical, merged V3+V4 spec — treat as both the substance/strategy authority; there is no separate competitor-research doc since this isn't a commercial business). Historical: [`docs/project-spec-v4-addendum.md`](docs/project-spec-v4-addendum.md) (superseded by project-spec.md, kept for reference only).

## Decisions made (do not revisit)

*Decisions 1–10 are pulled directly from `docs/project-spec.md`'s own "Decisions Resolved" (§9) plus earlier sections that already fixed a shape. Decisions 11+ are judgment calls made during the build where the spec was silent or self-contradictory, since the user asked for the build to proceed without stopping to ask — each is logged here, in code comments at the point of decision, and is a deliberate, loggable action to revisit if wrong.*

1. **Visual direction = "solar blue"** — full hex token table in spec §3. Set in `template/app/globals.css` (includes tier colors + the motion keyframes from §3).
2. **Tier system** — Ember (0–299) / Volt (300–599) / Current (600–999) / Plasma (1,000+), computed from `season_joules`, never stored. Spec §1/§4.
3. **Joule lifecycle** — `lifetime_joules` never resets; `season_joules` drives tier + leaderboard and resets on season end. Season cadence is admin-configurable (`Season` record: `start_date`/`end_date`/`cadence`), not hardcoded. Spec §2/§5.
4. **RBAC** — Owner/President (everything incl. Force Reset + Student Data Vault), Officer (Grid Management + Surge Builder, not the Vault), Event Volunteer (QR Scan Station only, scoped to one event, access auto-expires when it ends). Spec §7.
5. **JouleTransaction is the ledger of record** — season/lifetime totals are derived by summing it (also the audit trail), never a mutated balance column. Spec §5/§11.
6. **CSV question-import schema is final**: `question,option_a,option_b,option_c,option_d,correct_option,time_limit_seconds,tag`. Starter bank at `docs/starter-question-bank.csv` — not yet wired into a seed migration (it's real content, not a fixture, so import it through the Surge Builder's own CSV importer once there's a real Surge to attach it to).
7. **QR anti-abuse defaults** — token rotates every 90s during a live event; scan window is 15min before/after the scheduled time; geofence 150m as a **soft flag for admin review**, not a hard block; one scan per student per event. Spec §9. **Hardened beyond the spec's literal words**: spec says the repeat-scan check is "rejected client-side" — this build also enforces it server-side (a partial unique index on `joule_transactions`), since client-side-only would be a real hole in a security-first app.
8. **Concurrency plan for Surges** — pre-fetch the full question set once per Surge entry (no per-question polling); write answers to an append-only queue, not a locking balance update; compute the Surge Matrix reveal as one aggregation pass after the Surge closes, not live per-answer; rate-limit per student. Sized for 500 concurrent joiners within a 10s window (not load-tested — Postgres + RPC design supports it, but this environment can't spin up 500 concurrent clients to prove it). Spec §11.
9. **★ Season calendar — PLACEHOLDER, not real.** Seeded in `0007_jules_seed.sql` as two clearly-labeled placeholder rows (Even/Winter 2025-26, Odd/Monsoon 2026) so "today" always lands inside one. Replace via Institution Settings before the first real season.
10. **★ College email domain — PLACEHOLDER, not real.** `allowed_domains` (text array on `institution_settings`, singleton row), regex-equivalent-checked via `is_email_domain_allowed()`, editable from Institution Settings. Seeded to `yourcollege.edu`.
11. **`season_joules`/`lifetime_joules` are computed, not stored columns** — spec §5 lists them as Student fields, but §5/§11 also insists the ledger is the only source of truth ("never a mutated balance column" — decision 5). Those two statements conflict if taken as "store AND derive"; the stronger, more explicit rule wins. Resolved as SQL functions (`student_season_joules`, `student_lifetime_joules`) computed on read. Logged as a self-consistency fix, not a silent pick.
12. **Surge scoring is a flat `points_per_question` per Surge (default 20), not speed-weighted.** The spec never assigns a number to a correct Surge answer, only to event scans (10/25/50). A flat value was chosen because the spec's own Phase-2 tiebreaker ("average answer time, then earliest completion") only makes sense if raw totals tie often — which requires flat, not speed-scaled, scoring. Tunable per Surge at creation.
13. **Streak rule** (spec has `streak_days`/`last_active_date` fields but never defines the mechanic): increments once per calendar day on any Joule-earning action (event scan or correct Surge answer); no action for a full day resets it to 1 on the next action (not to 0 — it restarts the count rather than zeroing until next earn). Implemented in `_bump_streak()`.
14. **RBAC granularity the spec doesn't spell out**: `admin_manual_adjustment` and `role_change` (both named in the `AuditLogEntry.action` enum) are Owner-only, matching the Vault being Owner-exclusive. Officers cannot adjust Joules or change roles.
15. **A real cross-user data leak was found and fixed during the build**, not after: `student_season_joules()`/`student_lifetime_joules()` were initially left with Postgres's default PUBLIC execute grant, meaning any authenticated client could call them via RPC with an arbitrary student UUID and read someone else's private Joule totals. Fixed by revoking public execute on both and making `my_totals()` (the only legitimate caller for a student's own data) `SECURITY DEFINER` so it can still reach them internally while direct client calls are denied. This is exactly the class of bug the golden path's cross-user-denial gate exists to catch — logged here since it was caught by re-reading the migrations, not by running the gate against a live app (that gate still needs to be run for real once Docker is available).
16. **QR token is a deterministic HMAC over a 90-second time epoch, not a stored/mutable column.** Spec §5 lists `qr_code_token`/`qr_token_expires_at` as Event fields; this build computes the current token from `hmac(event_id || epoch, secret)` instead, so there's no cron to rotate it and no write contention across serverless instances — strictly stronger than the literal field design. `pgcrypto` + a server-only `app_secrets` table back it.
17. **Account recovery uses Supabase's built-in `resetPasswordForEmail` (an email magic-link) instead of a hand-rolled 6-digit OTP table.** Same user-facing shape the spec describes ("Lost connection?" → email → new password), fewer moving parts to get wrong.
18. **No QR-scanning camera library and no QR-image-rendering library were added** (the one-dependency rule: no new deps without asking). Instead: the QR the Grid Station displays encodes a deep link (`/scan?e=<event>&t=<token>`) that a student's native camera app opens directly — no in-app decoder needed — and the Station shows the current code as large text + a copyable link rather than a rendered QR image. Fully functional and testable end-to-end; wiring an actual scannable QR image is a flagged follow-up needing a one-line dependency decision (either vendor a small QR encoder or add a package).
19. **First Owner admin is bootstrapped via a standalone script** (`scripts/bootstrap-owner.ts`, run once via `npm run bootstrap:owner <email> <name>`), since the in-app admin-creation RPC is itself Owner-gated — a deliberate chicken-and-egg break, not an oversight. See `docs/runbooks/jules-setup.md`.
20. **`/admin/login` must never sit inside the auth-guarded admin layout** — it originally did (`app/admin/layout.tsx` wrapped every route under `app/admin/*`, including `login/`), so an unauthenticated visit to `/admin/login` hit `requireAdmin()`, got redirected back to `/admin/login`, and looped forever (`ERR_TOO_MANY_REDIRECTS`, caught live while verifying against the real DB, not by inspection). Fixed by moving every guarded route into `app/admin/(protected)/*` (a route group, same URLs) and leaving `app/admin/login/page.tsx` as an unguarded sibling. Applies generally: an auth guard's own login page can never live inside the layout that guard protects.
21. **Vercel project's team-wide "Standard Protection" (Vercel Authentication / SSO wall) was disabled for this project specifically**, via `PATCH /v9/projects/jules {"ssoProtection": null}` (no CLI command exposes this toggle). It was on by default (`all_except_custom_domains`), which would have put every deployment — including production — behind a Vercel login, meaning no student could ever reach the app. Confirmed via a live `curl`, which returned a `vercel.com/sso-api` redirect before the fix and a real `200`/`307` after. Revisit if the account's other projects rely on this protection for a reason.
22. **Vercel project's Framework Preset was stuck on "Other," not "Next.js,"** because it was created via `vercel project add` (empty project) rather than an auto-detecting first deploy. This made Vercel serve the app as a static site straight out of `public/` — the build itself succeeded (real Next.js build logs, correct route manifest) but nothing routed through it, so every real page 404'd even though the deploy reported "Ready." Fixed via `PATCH /v9/projects/jules {"framework": "nextjs"}` + a forced redeploy. **Lesson: "Ready" + a clean build log is not proof of a working deployment — a live route on the actual public URL is the only real gate**, exactly the golden path's own "works locally ≠ works live" rule (`docs/playbook.md` PART 5), just rediscovered at the infra layer instead of the app layer.
23. **Git-based auto-deploy-on-push is wired up and proven.** Bare `vercel git connect` failed because `template/` (the Vercel project root) isn't itself a git repository — it's a subfolder of the `jules/` repo — but `vercel git connect <explicit-https-url>` works around that, and `rootDirectory: "template"` was set via `PATCH /v9/projects/jules` (same API used for decisions 21/22; no CLI flag exposes it). Confirmed for real: a subsequent `git push` auto-triggered a production build that went `Building → Ready` in under a minute and aliased correctly.
24. **"Live Round" — a second, host-paced Kahoot-style quiz mode, alongside (not replacing) the original self-paced async Surge Mode.** Added after the user showed the app to a professor who asked for Kahoot-style group play with a live projector scoreboard. New tables (`live_rounds`, `live_round_teams`, `live_round_answers` — migration 0010) + RPCs (`host_create_round`, `join_live_round`, `host_advance_round`, `submit_live_answer`, `live_round_scoreboard`, `live_round_question` — migrations 0010/0011). Confirmed decisions from the user: **one phone per team** (no team-roster data model — one student hosts the team entry under their own login, Joules credit that student), **new mode alongside** the existing async one (different concurrency shape: host-paced/synchronous vs. self-paced/asynchronous), **live reveal after every question** (classic Kahoot), **one device for both host control and the projector display** (v1 scope — a separate anonymous spectator screen is a clean v2 add-on). Real-time is Supabase Realtime (`postgres_changes` on `live_rounds`/`live_round_teams`/`live_round_answers`, explicitly added to the `supabase_realtime` publication in 0010 — easy to forget, since RLS alone doesn't make a table broadcast). Routes: `/admin/live/new` + `/admin/live/[roundId]` (host), `/live` + `/live/[roundId]` (team).
25. **A real, previously-latent bug class: `RETURNS TABLE (id uuid, ...)` implicitly declares every output column (here, `id`) as a PL/pgSQL variable in scope for the WHOLE function body**, so a later unqualified `where id = ...` against a real table that also has an `id` column is genuinely ambiguous to Postgres. Found in `live_round_question()` (the Live Round team screen went blank after every phase change) — and, once every other `returns table` function was audited for the same shape, also in `start_surge()`, meaning **the original async Surge Mode has never been able to start a single Surge since it was built**; nobody had exercised it live until this session. Fixed by qualifying the table reference explicitly in both (migration 0012). Every other `returns table` function was checked and is safe — either no output column is literally named `id`, or every reference in the body was already alias-qualified (e.g. `admin_student_totals()` has an `id` output column but was always written as `s.id`, so it was never at risk).
26. **The Live Round team screen had a cold-load gap**: the scoreboard only re-fetched inside the Realtime UPDATE handler, so arriving directly at (or reloading into) the `reveal`/`leaderboard`/`complete` phase — not just transitioning into it live — rendered an empty scoreboard.     - **The streak itself is a full redefinition, not a rename**: it used to mean "opened the app today" (`_bump_streak()`, decision 13, bumped on every correct quiz answer or QR scan); it now means "how reliably has this student shown up for what they registered for." Computed on read via `student_attendance_streak()` — same "computed, never stored" philosophy as tiers/season_joules/lifetime_joules (decision 11) — by walking a student's already-concluded registrations backward from most-recent, counting consecutive attended ones until the first no-show. A no-show doesn't freeze the count, it resets it to 0 immediately; the *next* successful attendance starts counting from 1 again. A stored running counter would get this subtly wrong the moment two events conclude out of registration order; a query recomputed fresh every time cannot drift. `_bump_streak()` and its call sites in `submit_surge_answer`/`submit_live_answer` are removed entirely — quizzes no longer move the streak, only registered-event attendance does. The now-dead `students.streak_days`/`last_active_date` columns are dropped, and the field itself is renamed `streak_days` → `streak` everywhere it's returned (`my_totals`, `admin_student_totals`, and every frontend display) — it no longer counts calendar days, and keeping the old name would misrepresent what the number means now.
    - **Two real bugs, both the same class already seen in decision 46's migration** (Postgres's `CREATE OR REPLACE FUNCTION` refusing certain signature-adjacent changes) — `cannot change return type of existing function` when trying to rename `streak_days` → `streak` in `my_totals()`/`admin_student_totals()`'s return tables. Fixed the same way as before: explicit `DROP FUNCTION` first, and the migration's `down` path double-checked to make sure it drops-then-recreates in both directions rather than erroring the same way in reverse.
    - **Live-verified the actual logic, not just the schema**: since the Node routes are gated behind the mobile-PWA gate (decision 29) and this session's desktop browser tooling can't pass it, verification for this phase leaned on direct RPC/SQL tests against production instead of a screenshot — arguably the stronger proof for a business-logic feature like this one. Confirmed via `SET LOCAL request.jwt.claims`: `register_for_event` succeeds for a live not-yet-ended event and correctly rejects a second registration, an already-ended event, and (implicitly, via the `type <> 'surge'` check) a quiz. Confirmed `redeem_event_scan` both credits Joules and stamps the matching registration's `attended_at` in the same call. Confirmed `unregister_from_event` deletes an unattended registration but leaves an attended one alone. Confirmed the streak formula itself with a constructed attend→attend→**miss**→attend→attend→attend sequence: computed streak was `1` right after the miss (only the most recent attendance counted, the two before the miss did not), then `3` once two more consecutive attendances were added on top — exactly the "reset to 0, then recount from the next success" behavior the plan called for. ### Platform pivot, Phase 2 — Event registration + attendance-based streak
- **Built** (decision 47): migration 0021 — a new `event_registrations` table; `register_for_event()`/`unregister_from_event()` RPCs (reject surge-type events, already-ended events, double-registration; cancellation is blocked once attendance is stamped); `redeem_event_scan()` extended to stamp a matching registration's `attended_at` alongside its existing Joule credit, with walk-in scans (no prior registration) unaffected. The old daily-activity streak (`_bump_streak()`, decision 13) is removed entirely, along with its call sites in `submit_surge_answer`/`submit_live_answer` — quizzes no longer move the streak. A new computed function, `student_attendance_streak()`, replaces it: same "computed, never stored" philosophy as tiers/Joule totals (decision 11), walking a student's concluded registrations backward from most-recent and counting consecutive attended ones until the first no-show. The now-dead `students.streak_days`/`last_active_date` columns are dropped, and the field is renamed `streak_days` → `streak` everywhere (`my_totals`, `admin_student_totals`, and every frontend display — Profile, Dashboard, the Vault) since it no longer means "days," it means "consecutive attended events." Dashboard gained two new sections: "Upcoming events" (Register/Cancel buttons, a small client component using the same `useTransition`/server-action pattern already established in the Surge Builder) and "My registered events" (Registered/Attended/Missed status per row).
- **Two real Postgres migration bugs, both the same class already hit in decision 46's migration**: `CREATE OR REPLACE FUNCTION` refused to change `my_totals()`/`admin_student_totals()`'s return-table column name (`streak_days` → `streak`) with "cannot change return type of existing function" — fixed with an explicit `DROP FUNCTION` first, in both the `up` and `down` directions (the `down` path needed the same fix, not just the `up`). Worth remembering as a pattern now, not a one-off: this project's migrations have hit this exact Postgres restriction twice in two consecutive phases, for two different kinds of signature change (parameter rename in Phase 1, return-column rename in Phase 2) — any future `CREATE OR REPLACE FUNCTION` that changes anything about a function's declared shape, not just its body, should be treated as a likely `DROP FUNCTION` case from the start.
- **`npm run typecheck`/`lint`/`build` all passed clean** — one real lint catch: two inline `Date.now()` calls inside the Dashboard's render body (computing "is this event upcoming" and "has this registration concluded") tripped the project's own `react-hooks/purity` rule. Fixed with a standalone `hasConcluded()` helper, following the exact same pattern this same file's pre-existing `daysUntil()` helper already used for the season countdown.
- **Live-verified the actual logic directly against production**, not the UI (the Node routes sit behind the mobile-PWA gate, decision 29, same limitation as every earlier Node-route slice this session's desktop browser tooling hits). Applied migration 0021, confirmed via direct query that `streak_days`/`last_active_date` are gone and `event_registrations` has the right shape. Then, using `SET LOCAL request.jwt.claims`/`SET LOCAL ROLE authenticated` (same technique as Phase 1's RLS proof): confirmed `register_for_event` succeeds for a live not-yet-ended event and correctly rejects a second registration and an already-ended one; confirmed `redeem_event_scan` credits Joules and stamps `attended_at` on the matching registration in the same call; confirmed `unregister_from_event` deletes an unattended registration but leaves an attended one alone. Separately proved the streak formula itself with a constructed sequence of four/six past registrations (attend, attend, **miss**, attend[, attend, attend]) — computed streak was `1` right after the miss (correctly discarding the two attendances before it) and `3` once two more consecutive attendances were layered on top, exactly the "reset to 0, recount from the next success" behavior the plan called for.
